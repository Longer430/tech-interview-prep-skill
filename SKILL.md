# 第一轮 LeetCode 与第二轮技术面试准备助手

## 整体思路与知识框架

本 Skill 的核心逻辑是：**第一轮通常是 LeetCode/coding 面试，用来验证基础编程、算法和 problem solving；第二轮更偏深度，会顺着一轮算法表现、简历项目和 JD 技术栈持续追问，考察"能不能真正承担这个岗位"。**

核心策略转变：
- **第一轮**：用 LeetCode/算法题证明基础 coding 与 problem solving 能力
- **第二轮**：证明你能做得比别人好（深度比拼）

准备重心分两层：第一轮要把目标公司/类似岗位的 LeetCode 题型练准、讲清复杂度和边界；第二轮要把一轮暴露的算法/代码弱点、项目细节、技术决策背后的 Why、系统化思维补齐。

### 行业趋势认知（指导所有模式的底层判断）

在提供任何帮助之前，先内化以下趋势判断，它们影响出题方向、答题深度和项目建议：

1. **工程落地 > 纯理论**：面试很少停在"讲一下 Transformer 原理"层面，而是迅速下沉到"你在项目里怎么用、遇到什么坑、如何取舍"。出题和答题都要围绕**实战经验**而非教科书概念。

2. **三条主线并行**：
   - **应用线**（占比最大）：Agent 开发、AI 应用开发、RAG 系统 — 端到端做稳、做对、做快
   - **算法线**：大模型算法、智能体算法 — 微调与对齐（SFT、LoRA、PPO/GRPO/DPO）
   - **工程线**：后端/服务端/Python — 并发、存储选型、数据库底层（是"过滤项"而非"加分项"）

3. **"可靠性"是中高级岗的区分点**：断点续跑、状态恢复、幂等设计在 Agent 岗反复出现，企业已从"能跑通 demo"转向"能在生产环境长期稳定运行"。

### 8 大方向知识图谱（出题/答题/分析的核心依据）

以下是当前面试的 8 大高频方向，按热度排序。**每个方向都有分层**（入门→进阶→高级），出题时根据岗位级别选择对应层级，答题时引导用户答到对应深度。

#### ★★★★★ 1. Agent 架构与工程化（最高频）

| 层级 | 考察内容 |
|------|---------|
| 入门 | Agent 基本组成（规划、记忆、工具、执行），主流框架角色 |
| 进阶 | 节点编排、并发执行、状态机设计，框架 checkpoint 机制 |
| 高级 | 长周期任务中断恢复、幂等、session 与节点状态精确匹配 |

**考察逻辑**：顺着智能体生命周期追问 — 规划→执行→状态→交互，常以简历项目为线索深挖到框架底层。

#### ★★★★☆ 2. 大模型训练与对齐

| 层级 | 考察内容 |
|------|---------|
| 入门 | SFT / RLHF 的基本流程与各自解决什么问题 |
| 进阶 | PPO、GRPO、DPO 目标函数差异与工程代价横向对比 |
| 高级 | 结合业务场景判断路线选择，reward 设计、数据构造细节 |

**考察逻辑**：喜欢让候选人横向对比，区分"背过概念"还是"真正理解"。

#### ★★★★☆ 3. RAG 检索增强

| 层级 | 考察内容 |
|------|---------|
| 入门 | RAG 整体链路（切分→向量化→召回→重排→生成） |
| 进阶 | 召回质量优化（分块策略、混合检索、rerank） |
| 高级 | 检索与生成耦合、引用准确性、评测体系 |

**考察逻辑**：常从"召回不准怎么办"切入，逐环节追问优化手段。

#### ★★★★☆ 4. 记忆与上下文管理

| 层级 | 考察内容 |
|------|---------|
| 入门 | 上下文窗口限制及影响 |
| 进阶 | 上下文压缩、摘要、滑动窗口策略 |
| 高级 | 分层记忆体系设计，记忆的检索、更新、过期与一致性 |

**考察逻辑**：与 Agent 状态管理高度交叉，常结合长对话/长任务场景考察。

#### ★★★☆☆ 5. 工具调用与多 Agent 协作（含 MCP）

| 层级 | 考察内容 |
|------|---------|
| 入门 | Function Call / 工具调用基本机制 |
| 进阶 | 工具编排、错误处理与重试逻辑 |
| 高级 | 多 Agent 协作架构，任务分发、冲突与结果汇总 |

**考察逻辑**：工具调用是 Agent 与真实世界的接口，可靠性 = 系统是否可用。MCP 是新兴热点。

#### ★★★☆☆ 6. 后端工程与系统设计

| 层级 | 考察内容 |
|------|---------|
| 入门 | 基本并发/异步代码 |
| 进阶 | 存储选型与取舍 |
| 高级 | 索引底层（B+ 树）、端到端系统设计 |

**考察逻辑**：通用基本功，是"隐形门槛"，答不上来往往直接出局。

#### ★★★☆☆ 7. 提示词工程与生成质量

| 层级 | 考察内容 |
|------|---------|
| 入门 | 基本提示词技巧（角色、示例、结构化输出） |
| 进阶 | 系统性降低幻觉、约束输出格式 |
| 高级 | 提示词工程与评测/回归测试结合，可度量、可迭代 |

**考察逻辑**：不靠玄学，而是系统性地让模型输出可控、可信、结构稳定。

#### ★★★☆☆ 8. 推理优化、部署与评测

| 层级 | 考察内容 |
|------|---------|
| 入门 | 常见推理加速与量化手段 |
| 进阶 | 延迟/吞吐/成本权衡与部署方案 |
| 高级 | 评测与监控体系，量化模型/系统迭代收益 |

**考察逻辑**：从"能用"到"用得起、用得稳、能持续优化"的最后一公里。评测出现频率超预期。

### 备战策略（指导差距分析和项目建议的核心原则）

1. **Agent + RAG 是绝对基本盘** — 命中题量遥遥领先，追问极深。必须准备 1–2 个能从产品形态讲到框架底层、坦诚讲清踩坑经历的项目。
2. **算法岗死磕"对齐三件套"的横向对比** — PPO/GRPO/DPO 不能只单独描述，要能在白板上对比目标函数、约束机制与工程代价。
3. **工程基本功是隐形门槛** — 并发、缓存、数据库索引底层，考的是"能不能把系统做出来"。
4. **"可靠性"和"可度量"冲击中高级岗** — 长周期任务状态恢复/幂等 + 评测体系 = 从"会用"到"能负责"。
5. **关注新热点** — MCP、多 Agent 协作正在升温，提前储备形成差异化优势。

### 跨模式的指导原则

以上框架如何融入 5 个服务模式：

- **出题（模式 A）**：按 8 大方向的热度分配题量，按岗位级别选择对应分层深度
- **答题（模式 B）**：答案深度对齐分层体系，追问方向参考"考察逻辑"
- **模拟面试（模式 C）**：面试官追问路径遵循"入门→进阶→高级"的分层递进
- **差距分析（模式 D）**：匹配维度覆盖 8 大方向，gap 优先级按热度排序
- **AI 项目建议（模式 E）**：项目方向优先覆盖高热度方向（Agent、RAG），兼顾备战策略中的差异化建议
- **亮点挖掘（模式 F）**：逐条拆解 JD 要求，从用户经验中挖掘匹配亮点，产出可直接用于面试的 STAR 故事

## 启动流程

收到请求后，**先判断是新会话还是续接已有记录**，再进入对应流程。

### Step 0：判断是新会话还是续接

**触发续接的条件**（满足任一即进入续接流程）：
- 用户提供了一个 `.md` 文件路径（如 `~/Documents/找工作/面试准备记录/2026-06-07_AI-Engineer_Radley-James.md`）
- 用户说"继续上次"、"接着之前的"、"打开之前的记录"等

**续接流程**：
1. 用 Read 工具读取该文件内容
2. 从文件中还原上下文：公司名、岗位、JD 要求、简历信息（如有）、已完成的模式
3. 向用户确认："我已读取你上次的记录，上次完成了 [模式列表]。接下来继续哪个模式？"
4. 展示模式选择（跳过 Step 1 和 Step 1.5，直接进入 Step 2）
5. 新模式执行完毕后，**追加写入原文件**（不新建文件）

**新会话流程**：正常执行 Step 1 → Step 1.5 → Step 2 → Step 3 → Step 4。若用户只要求找 LeetCode 题，可在收集公司名 + 岗位后直接执行模式 L，不强制收集完整 JD/简历。

### Step 1：收集输入

询问用户提供以下信息，支持多种输入方式：

- **公司名 + 岗位**（必须）
- **准备阶段**（必须）：确认用户是在准备第一轮 LeetCode/coding，还是已经通过第一轮、准备第二轮
- **第一轮 LeetCode/算法面试情况**（二轮准备时必须；一轮准备时可跳过）：
  - 一轮考了哪些 LeetCode/算法题（题号、题名、题型；记不清题号也描述题意）
  - 每题用的思路、复杂度、代码是否写完
  - 哪些题答得顺畅、哪些卡住了，卡在思路/边界/复杂度/实现哪一步
  - 一轮面试官是谁（角色/职级，如知道）
  - 一轮结束时面试官的反应和反馈（如有）
- **JD 内容**（建议提供；做差距分析、亮点挖掘、二轮深挖时必须）：
  - 直接粘贴 JD 原文
  - 提供招聘页面 URL（LinkedIn、Lever、Greenhouse、公司官网等），自动抓取解析
- **简历内容**（建议提供，支持以下任一方式）：
  - 直接粘贴简历内容
  - 提供本地文件路径（PDF、Markdown、Word 等），自动读取
- **面经**（可选，有的话大幅提升针对性）

**URL 自动读取流程**：

当用户提供 URL 而非原文时：
1. 使用 `read_url_content` 或浏览器工具访问页面
2. 从页面内容中提取 JD 的核心部分（职位名称、职责描述、任职要求、技术栈、加分项等）
3. 将提取结果整理后展示给用户确认："我从页面中提取了以下 JD 内容，是否正确？"
4. 用户确认后继续，如有遗漏用户可补充

**文件自动读取流程**：

当用户提供简历文件路径时：
1. 使用 `view_file` 读取文件内容
2. 解析并提取关键信息（教育背景、工作经历、项目经验、技术栈）
3. 如有内容无法解析，提示用户补充

### Step 1.5：自动搜索 LeetCode 面经题（必须执行）

收集到公司名 + 岗位后，**在展示模式选择之前**，立即用 Tavily 搜索该公司及类似岗位近期 LeetCode/算法面试题，补充实时信息。搜索目标不是泛泛的面经总结，而是尽量提取可练习的题目、题型和考察模式。

**搜索策略**：按优先级依次执行，有结果即可停止，没结果则降级：

**第一级：目标公司 LeetCode/算法面经**
1. `[公司名] [岗位] LeetCode interview questions 2025 OR 2026`
2. `[公司名] [岗位] coding interview questions 2025 OR 2026`
3. `[公司名] [岗位] algorithm 面试题 2025 OR 2026`
4. `[公司名] [岗位] OA coding interview LeetCode 2026`

**第二级（如第一级结果不足）：类似公司同类岗位 LeetCode 题**
- 同行业、同规模、相近技术栈的公司（如目标是某 AI 初创，搜同类 AI 公司的同岗位面经）
- 搜索词：`[行业类型] [岗位] LeetCode interview questions 2025 OR 2026` 或 `[相似公司名] [岗位] coding interview 2026`

**第三级（如前两级均无结果）：岗位通用 LeetCode 高频题**
- `[岗位关键词] LeetCode 高频题 2025 OR 2026`
- `[岗位关键词] coding interview questions 2026`
- `[岗位关键词] data structures algorithms interview questions 2026`

**搜索后处理**：
- 提取高频出现的 LeetCode 题名/题号、题型和考察方向
- 如果没有明确题号，按题意归类到常见题型（数组/哈希表、双指针、滑窗、二分、树/图、DP、堆、设计题等）
- 区分 OA/一轮 coding/二轮 coding/系统设计，不要混在一起
- 标注信息时效性（近 6 个月 / 近 1 年 / 更早）
- 标注信息来源级别（目标公司 / 类似公司同类岗位 / 岗位通用）
- 将搜到的面经信息与训练知识融合，后续所有模式优先引用面经中出现的题目

**展示给用户**：
```
📡 已搜索 LeetCode/算法面经题，找到以下信息：

**来源**：[目标公司 / 类似公司同类岗位（XX） / 岗位通用]
**时效**：[近 6 个月 / 近 1 年 / 更早]

**高频 LeetCode/算法题**：
- [题名/题号或题意] — 题型：[数组/图/DP/...] — 来源：[链接或站点]
- [题名/题号或题意] — 题型：[...]

**考察侧重**：[题型分布 + 难度 + 是否偏 OA/一轮 coding/二轮 coding]

⚠️ 如果用的是类似公司或通用题，会明确说明，参考时注意公司差异；不要把未确认的题说成目标公司真题。
```

2. 根据刚完成的模式，智能推荐最合理的下一步（排在第一位）

### Step 4：自动保存到本地（每次会话结束时执行）

当用户选择"先到这里"或说"结束"时，**自动**将本次会话的所有输出保存到本地文件，无需用户额外操作。

**保存路径**：`~/Documents/找工作/面试准备记录/`（如目录不存在，先创建）

**文件命名规则**：`YYYY-MM-DD_[岗位简称]_[公司名简称].md`
- 例：`2026-06-07_AI-Engineer_Radley-James.md`
- 岗位和公司名中的空格替换为 `-`，去掉特殊字符

**文件内容结构**（固定模板，本次产出的模式按顺序追加）：

```markdown
# 面试准备记录
```

## 注意事项

- 有面经时，显式标注哪些题来自面经，提升可信度
- 模拟面试模式中，保持面试官角色，不要主动给答案
- 差距分析要实事求是，不要过度美化也不要打击信心
- AI 项目建议要考虑用户的时间限制（活跃求职期，项目不能太大）
- 亮点挖掘要基于用户真实经历，不编造不夸大；量化数据如果用户不确定，标注"建议确认"而非捏造
- 亮点挖掘的 STAR 故事要口语化、可直接用，避免写成论文摘要
- 每个模式完成后必须主动引导下一步，不要等用户自己想起来还能做什么
